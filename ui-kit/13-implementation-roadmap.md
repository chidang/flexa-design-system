# 13 — Implementation Roadmap (Bootstrap Decisions + U0→U12)

> Companion to the 12 SSOT documents — this is the **execution plan**, not a design document. It locks the five bootstrap decisions the design docs deliberately left open, and sequences the build into slices. An AI agent (or human) starting from zero reads: `README.md` → `12-prompt-library.md` (routing) → **this document** → picks the next open slice.
>
> Rule of precedence: if this document ever contradicts docs 01–11, **docs 01–11 win** — fix this file. Code follows docs; a behavior change lands as a doc PR first (or in the same PR).

---

## 1. Bootstrap decisions (locked)

### D-1. Where the code lives

- **Package:** `packages/ui` in this monorepo, package name = npm publish name **`flexa-ui`** (bare name — the `@flexa` npm scope is taken, so the package name IS the publish name, exactly as `flexa-design-system` resolved the identical scope problem at FDS-2, doc 19 QĐ-19.7). Keeping one name means JS (`flexa-ui`) and CSS (`flexa-ui/styles.css`) resolve through the same specifier in the workspace and when published.
- **Dependency direction (one-way, never reversed):** `flexa-ui` → `flexa-design-system` (FDS). `flexa-ui` NEVER imports `@flexa/core`, `@flexa/editor`, or anything builder-specific. The builder does not import `flexa-ui` either — they are siblings on top of FDS.
- **Publish shape mirrors FDS-2 (doc 19 QĐ-19.8):** dev exports `src/`, publish exports `dist/` (ESM + d.ts) via `publishConfig`; CSS artifacts as subpath exports (`flexa-ui/styles.css`, per-component files under `flexa-ui/css/*`).
- Peer dependencies: `react >= 18`, `react-dom`. No zod, no CSS-in-JS runtime, no utility-class framework (inherits FDS anti-goals).
- **Docs consume flexa-ui:** the public docs site `apps/fds-docs` (fds.sitebefy.com) gains a **Components** section that depends on `flexa-ui` (`workspace:*`). Consequence: the public extraction to `flexa-design-system-public` — which today carries `packages/fds` — must **now also carry `packages/ui`**, mirroring how FDS flows builder → public → npm one-way. See §5.

### D-2. Framework & CSS strategy

- **First target: React 18+ function components + TypeScript.** Rationale: every current Flexa product surface (Next.js apps, WP admin React) is React. The **portable layer is the markup + CSS contract** from doc 04 (`.fx-*` classes, `data-variant/size/tone`, `.is-*` states) — a future Vue/web-component implementation re-uses the same CSS and class contract unchanged.
- **Styling: plain CSS files, one per component** (`src/button/button.css` → `.fx-button { … }`), consuming only `var(--fx-*)`. No CSS-in-JS, no Tailwind, no preprocessor. Theming/dark mode work by construction (semantic vars, `data-fx-scheme`) — components contain zero scheme logic.
- Component API = doc 04 contract verbatim: controlled/uncontrolled per 04 §1, event names per 04 §1, every baked-in string is a prop with a documented default (i18n rule).

### D-3. Icons

- **Bundled set: Lucide** (`lucide-react`, MIT) wrapped in **`FxIcon`** with a canonical-name map: docs reference icon names (`home`, `package`, `scale`, `bell`, …) — the map (`src/icon/map.ts`) translates canonical name → Lucide glyph, so the underlying library is swappable without touching docs or call sites.
- Sizes 16/20/24 only (foundations §icons); icon-only controls require `aria-label` (gate U0 enforces).
- The canonical name list is seeded from every icon name used in docs 02/06/08 — the map file is the SSOT for available names; a missing name is a build error, not a silent fallback.

### D-4. Mock backend

- **MSW (Mock Service Worker) handlers generated from doc 09**, living at `packages/ui/mocks/` with subpath export `flexa-ui/mocks` (`msw` is a devDependency of the workspace, optional peer for consumers).
- Fixture factories follow prompt P10 (doc 12): valid enums from 04 §5, ULID ids, `{amount, currency}` Money objects, ISO-8601 UTC dates. Deterministic by default (seeded), so kitchen-sink screenshots and tests are stable.
- Scope: only the endpoints the reference screens (U11) need — grow on demand; every handler cites its 09 section in a comment.

### D-5. CI gates (machine-enforced, not model-self-check)

All gates are vitest specs in `packages/ui/tests/`, running in the existing `pnpm test` + typecheck pipeline:

| Gate | What it rejects |
|---|---|
| `token-discipline.spec.ts` | Any component CSS containing hex/rgb()/hsl() literals, or `px` values outside the canonical allowlist (control heights 32/40/48, touch 44, toast 360, drawer 400, sidebar 240/64, top bar 56, dialog 440, breakpoints — README + 02/04 canonical tables, kept as an exported const) |
| `enum-drift.spec.ts` | Code enums (`PaymentStatus`, `EscrowStage`, `OrderStatus`, `ListingStatus`, `DisputeStatus`, tone/size/variant unions) differing from doc 04 §5 canonical values (spec holds the canonical arrays verbatim) |
| `a11y.spec.ts` per component | axe violations (jsdom + testing-library + jest-axe) on every state variant rendered in tests — necessary-not-sufficient per doc 11 §testing; manual SR pass stays in the release checklist |
| `class-contract.spec.ts` | Rendered markup missing the root `.fx-<name>` class or using className modifiers where `data-*` attributes are contracted (04 §1.2) |
| `icon-map.spec.ts` | Canonical icon names referenced in code but absent from the map |

Visual workbench: **`apps/ui-kitchen-sink`** (Vite app, mirrors `apps/playground` — no Storybook dependency): one route per component rendering **all 10 states × variants × sizes**, plus a scheme/density/brand toggle bar (reuses `applyBrand` + `emitTheme` from FDS like `apps/fds-docs`). This is the visual reference that substitutes for Figma; screenshots from it accompany every slice PR.

## 2. Working agreement (every slice)

- **Slice = branch = PR** off `main` (or stacked), one reviewable unit, matching repo convention (doc 20 §6 culture).
- Per-component Definition of Done:
  1. Props/events/keyboard/ARIA exactly per its 04 contract (full or condensed) — deviations require a 04 doc change in the same PR.
  2. CSS token-only, states per README vocabulary, responsive per 02.
  3. **Showcase spec** `src/<name>/<name>.showcase.ts` authored (variants = all applicable states × variants × sizes; `props`/`events`/`keyboard`/`aria` transcribed from the 04 contract; `enums` referencing the component's **own** exported unions, never re-typed; `contract` anchor into the markdown SSOT; `category` + `slice`; `interactive: true` if it uses hooks) **and registered** in `src/registry.ts` (one import + one array entry). The kitchen-sink page **and** the fds-docs Components page both auto-derive every state from this single source — no separate kitchen-sink authoring, no nav edit. See §5.
  4. Tests: behavior (testing-library) + keyboard map + axe per state.
  5. All five gates green; typecheck green.
  6. **Docs prose** `apps/fds-docs/content/components/<name>.json` (~5 lines: `tagline` + optional `whenToUse`/`blocks`) — the only docs-app edit per component; index + nav populate from the registry.
  7. **Docs gates green:** `showcase-registry.spec.ts` (slug-unique, `enums` == component unions, `contract.anchor` resolves in the SSOT) + `apps/fds-docs` typecheck + `next build` static-export of the new `[slug]` route.
- AI agents implement components with prompt **P1** (doc 12), pasting the component's 04 contract + 01 §4 + 11 quick card; screens with **P2**. Unknown/ambiguous contract → `TODO(ssot)` + stop (doc 12 §5), never improvise.
- Component order within a slice: dependencies first (e.g. Field Group before inputs that compose it).

## 4. What stays out of v1 (anti-goals)

- No second framework target (Vue/web components) until React ships — the CSS/class contract is the forward-compat layer.
- No chart library, no rich-text/markdown editor engine decision (Rich Text Editor / Markdown Editor ship as contract-compliant shells around a pluggable editor slot; revisit on demand).
- No Storybook, no visual-regression SaaS in v1 — kitchen-sink + PR screenshots; add Playwright screenshot tests only when churn justifies it.
- No design-token additions from this package — token gaps go to FDS as proposals (01 §4.4).
- No app state management, routing, or data-fetching opinions — components take data via props; MSW mocks exist for the workbench, not as a client library.

## 5. Documentation integration — fds-docs "Components" section

The public component reference is a **new section of the existing docs site `apps/fds-docs`** (fds.sitebefy.com), not a separate site — it reuses that site's proven build-time-generator architecture (`src/tokens.ts`/`recipes.ts`/`packs.ts`, the `src/content.ts` block seam, `Playground.tsx`'s `applyBrand`→`emitTheme` scoped-CSS toggle, `blocks.tsx`, `CopyButton.tsx`). Authored in the builder, extracted to `flexa-design-system-public` like FDS (D-1). Built **incrementally**: one component's docs land in its own slice PR (U1→U12), so docs always match shipped code.

**Single source of truth for demos — the showcase spec.** Each component exports a `ShowcaseSpec` (pure data: `variants` = states × variants × sizes, `props`/`events`/`keyboard`/`aria` transcribed from doc 04, `enums` referencing the component's **own** unions, `contract` = `{ doc, anchor }` into the markdown SSOT, `category`, `slice`, `interactive?`) co-located at `packages/ui/src/<name>/<name>.showcase.ts` and aggregated into `FLEXA_UI_COMPONENTS` in `packages/ui/src/registry.ts`. **Both** the kitchen-sink workbench (D-5) **and** the docs Components page derive every demo from this one registry — there is no competing docs-only demo mechanism, and the registry doubles as the inventory-completeness list (rides `enum-drift.spec.ts`).

**Generated reference, linked contract.** The docs component page is generated from the spec (live demo grid + props/events/keyboard/ARIA tables + canonical enum chips + a **"full contract →"** deep-link to the markdown SSOT anchor). Deep prose is **not** ported into block-JSON — the bible docs (04/05/11…) stay SSOT and are linked. The only hand-authored per-component docs artifact is a ~5-line `apps/fds-docs/content/components/<name>.json` (`tagline`/`whenToUse`). Index + nav auto-populate from the registry, so no per-slice nav edits.

**Static-export discipline.** Non-interactive components render as RSC to static HTML at export time (no client JS); only hook-driven components (Toast/Dialog/Tabs/Select/…) mount in a `"use client"` island (`ComponentDemo.tsx`), with body-level portals contained to a scoped container. flexa-ui's per-component CSS is injected via a nested `app/components/layout.tsx` importing `flexa-ui/styles.css` **scoped** to the section so `.fx-*` rules never leak into the docs chrome; theme vars stay scoped via `[data-fx-theme="components"]`.

The one-time wiring lives in **U0**; from U1 on, each component adds only DoD steps 3 + 6 (§2).
